# Admin Tool Data Contract

Version: 2026-04-21

## Purpose

This document defines how Herman Portal must use the tenant-owned identity, invitation, and branding data written by Herman Admin.

Herman Admin is the system of record for:

- tenant portal configuration
- invitation issuance and invitation status
- tenant-scoped onboarding and admin-managed setup state

Herman Portal is the system of execution for:

- login
- invitation acceptance
- initial password setup
- password reset
- session creation
- redirect into HermanPrompt

## Shared Data Ownership

### `auth_users`

`auth_users` is the source of truth for portal-authenticated users.

Portal requirements:

- authenticate against the user identified in `auth_users`
- use `email` as the login identifier
- use `user_id_hash` as the stable identity passed into HermanPrompt launch tokens
- do not create a parallel identity store outside `auth_users`

### `tenant_portal_configs`

This table is written by Herman Admin and read by Herman Portal.

Required columns:

- `tenant_id`
- `portal_base_url`
- `logo_url`
- `welcome_message`
- `is_active`

Portal requirements:

- when tenant context is known, load the matching `tenant_portal_configs` row
- use `logo_url` to render the tenant logo on `/login` and `/invite`
- use `welcome_message` to render tenant-specific welcome copy on `/login` and `/invite`
- if no row exists, or branding fields are null, fall back to the default portal branding
- respect `portal_base_url` as the expected tenant-facing base URL for invite acceptance links generated by Herman Admin

### `user_invitations`

This table is written by Herman Admin and read by Herman Portal.

Required columns:

- `user_id_hash`
- `tenant_id`
- `email`
- `invite_token_hash`
- `status`
- `expires_at`
- `accepted_at`
- `revoked_at`
- `provider_message_id`
- `last_error`

Portal requirements:

- `/invite` must hash the raw token from the query string and match it against `invite_token_hash`
- Herman Admin generates raw invitation tokens with `secrets.token_urlsafe(32)`
- Herman Admin stores `invite_token_hash` as the lowercase SHA-256 hex digest of the raw token
- the portal must only accept invitations whose `status` is active for acceptance, such as `pending` or `sent`
- the portal must reject invitations whose `expires_at` is in the past
- the portal must reject invitations with `revoked_at` populated
- successful acceptance must mark the invitation `accepted` and set `accepted_at`
- the portal must not store raw invitation tokens in the database

### `auth_user_credentials`

This table will be owned by Herman Portal and read by Herman Portal.

Required columns:

- `user_id_hash`
- `password_hash`
- `password_algorithm`
- `password_set_at`
- `failed_login_attempts`
- `locked_until`
- `last_login_at`

Portal requirements:

- `/login` verifies password against `password_hash`
- `/invite` creates or updates the first credentials row when the user sets their initial password
- passwords must only be stored as secure hashes

## Runtime Behavior

### Standard login flow

1. User opens Herman Portal `/login`.
2. Portal resolves tenant context if available.
3. Portal loads `tenant_portal_configs` for the tenant if present.
4. Portal renders tenant logo and welcome message if configured.
5. User signs in with email and password.
6. Portal reads `auth_users` and `auth_user_credentials`.
7. Portal signs the HermanPrompt launch token and redirects into HermanPrompt.

### Invitation acceptance flow

1. User opens Herman Portal `/invite?token=<raw-token>`.
2. Portal computes `sha256(raw_token).hexdigest()` and matches it against `user_invitations.invite_token_hash`.
3. Portal looks up the corresponding `user_invitations` row.
4. Portal verifies:
   - invitation exists
   - invitation status is acceptable for first use
   - invitation is not expired
   - invitation is not revoked
5. Portal loads the tenant row and `tenant_portal_configs` using `tenant_id`.
6. Portal renders:
   - tenant logo from `logo_url` when present
   - tenant welcome message from `welcome_message` when present
   - invited email address
7. User sets initial password.
8. Portal writes `auth_user_credentials`.
9. Portal marks the invitation accepted.
10. Portal activates the user if needed and creates the session / launch token.
11. Portal redirects the user into HermanPrompt.

## Branding Rules

Branding scope for this phase is limited to logo only plus welcome message content.

Portal must support:

- logo override from `tenant_portal_configs.logo_url`
- welcome message override from `tenant_portal_configs.welcome_message`

Portal must not assume support for:

- custom theme colors
- custom fonts
- custom layouts

## Fallback Rules

If Herman Portal cannot resolve a tenant portal config row, it must fall back to:

- default platform logo
- default welcome message
- default portal base URL

If Herman Portal cannot resolve an invitation token, it must show a safe invalid-invitation state without exposing sensitive details.

## TTL Rules

Invitation tokens expire after 7 days.

Portal must treat `expires_at` as authoritative.

Portal must not calculate a different TTL independently once `expires_at` exists in the shared database.

## Implementation Note

Herman Admin already owns the data-entry experience for:

- portal base URL
- logo URL
- welcome message
- invitation generation and send

Herman Portal should therefore focus on consuming those persisted values rather than introducing a second admin surface for the same settings.
