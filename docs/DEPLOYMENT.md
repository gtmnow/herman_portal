# Deployment Guide

## Overview

Herman Portal deploys as two Railway services:

- frontend service
- backend service

The portal is the production login entrypoint for HermanPrompt and the execution layer for invitations created by Herman Admin.

## Service Layout

### Frontend service

- Root directory: `frontend`
- Purpose: serve the login, invite, forgot-password, reset-password, and change-password UI

### Backend service

- Root directory: `backend`
- Purpose: authenticate users, accept invitations, issue signed launch tokens, manage password reset/change flows

## Shared Database Contract

Herman Portal must be deployed against the same shared Postgres database used by HermanPrompt and Herman Admin.

Portal deployment assumes the shared database contains:

- `auth_users`
- `tenant_portal_configs`
- `user_invitations`
- portal-owned auth credential/session tables

For the detailed contract, see:

- `docs/ADMIN_TOOL_DATA_CONTRACT.md`

## Railway Backend Configuration

### Build and start

- Build command:

```bash
pip install -r requirements.txt
```

- Start command:

```bash
python -m alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

### Required environment variables

- `DATABASE_PUBLIC_URL=<shared Postgres url used by HermanPrompt and Herman Admin>`
- `HERMANPROMPT_UI_URL=https://herman-prompt-demo-production-5b99.up.railway.app`
- `PORTAL_UI_URL=https://hermanportal-production.up.railway.app`
- `HERMANPROMPT_LAUNCH_SECRET=<must exactly match HermanPrompt backend AUTH_LAUNCH_SECRET>`
- `LAUNCH_TOKEN_TTL_SECONDS=3600`
- `PASSWORD_RESET_TOKEN_TTL_SECONDS=1800`
- `INVITATION_TOKEN_FALLBACK_TTL_SECONDS=604800`
- `DEV_SHOW_RESET_LINKS=false`
- `CORS_ALLOWED_ORIGINS=https://hermanportal-production.up.railway.app`
- `APP_ENV=production`
- `HOST=0.0.0.0`

Notes:

- the backend accepts either `DATABASE_URL` or `DATABASE_PUBLIC_URL`
- `postgres://...` and `postgresql://...` are normalized to the installed `psycopg` driver automatically
- because the database is shared with other services, Herman Portal uses its own Alembic version table: `alembic_version_herman_portal`
- invitation acceptance should rely on `user_invitations.expires_at` when present; the fallback TTL env is only for compatibility during migration

## Railway Frontend Configuration

### Build and start

- Build command:

```bash
npm install && npm run build
```

- Start command:

```bash
npx serve -s dist -l $PORT
```

### Required environment variables

- `VITE_API_BASE_URL=https://hermanportal-production-8c42.up.railway.app`

Important:

- `VITE_*` variables are read at build time
- changing `VITE_API_BASE_URL` requires a new frontend deploy

## Production Flow

### Login
1. user opens Herman Portal frontend
2. portal renders tenant branding when tenant context is known
3. user logs in with email and password
4. Herman Portal backend validates credentials against the shared Postgres database
5. Herman Portal backend signs launch token
6. Herman Portal frontend redirects user into HermanPrompt frontend
7. HermanPrompt backend validates the launch token and creates the app session

### Invitation acceptance
1. user opens invite link generated by Herman Admin
2. Herman Portal frontend loads `/invite?token=...`
3. Herman Portal backend validates the token against `user_invitations`
4. Herman Portal backend loads tenant logo and welcome message from `tenant_portal_configs`
5. user sets initial password
6. Herman Portal backend stores credentials and marks invitation accepted
7. Herman Portal redirects user into HermanPrompt

## Local And Production Data Setup

### Run migrations

```bash
cd backend
python -m app.scripts.init_db
```

### Data expectations

Before invitation acceptance can work end to end, Herman Admin must already have written:

- a tenant row
- a matching `tenant_portal_configs` row or defaultable absence
- an `auth_users` row for the invited user
- a `user_invitations` row with a valid `invite_token_hash` and `expires_at`

## Common Deployment Issues

### Portal login works but invite flow does not

Common causes:

- `user_invitations` table missing required TTL/status fields
- invite token hash comparison is not aligned with the admin generation logic
- tenant branding lookup is querying the wrong tenant id

### Frontend shows default branding instead of tenant branding

Common causes:

- no `tenant_portal_configs` row exists for the tenant
- `logo_url` and `welcome_message` are null
- portal failed to resolve tenant context from the invitation or login flow

### Backend fails during migration

Common causes:

- shared database already has another service’s `alembic_version` table
- missing `DATABASE_PUBLIC_URL` or `DATABASE_URL`
- unsupported connection string format

Fix:

- use the portal’s Alembic setup, which writes to `alembic_version_herman_portal`
- set `DATABASE_PUBLIC_URL` or `DATABASE_URL` on the backend service
- confirm the URL points to the shared Postgres database

## Production Verification Checklist

1. portal login page loads
2. valid login redirects into HermanPrompt
3. invite link opens `/invite`
4. `/invite` displays tenant-specific logo and welcome message when configured
5. expired invite is rejected
6. invitation acceptance sets the initial password successfully
7. forgot-password flow returns accepted response
8. change-password flow succeeds
9. new password works and old password fails
